Automated Acceptance Tests with Behat - Session Blog

Peter McClory
Tuesday, 24th September, 2013

The following is a live-stream (in chronological order) hosted by Craig and Pete that took place during the ‘Automated Acceptance Test with Behat’ session at DrupalCon Prague on Tuesday 24th September 2013. Due to the internet being flaky at the event, we were unable to publish a lot of this data in real-time.

Nathan: Behat - it's all about delivery.

Nothing can compensate for project failure. If a football team loses, there's nothing that can be done. They've lost. How can we best pre-plan to minimise the chance for failure?

Nice quote from Nathan - People don't want to buy a quarter-inch drill. They want a quarter-inch hole. - Theodore Levitt
-- 13:03

Developers love to solve problems, but they must be set the correct challenge, so that they are aware of what they are tackling.
-- 13:05

Communication is key, within the entire team. Developers cannot be left to describe what they think they should do, as often they will define something that is not in line with project deliverables, and this will generally result in project failure of a certain level at least.
-- 13:09

Feature Declaration:

As a [User]
I want [Feature]
So that [Benefit]
-- 13:10

Clients should be encouraged to "accept" specific behaviour from features, as described by them in a format that is standardised.

If a feature behaves in [the way described here] then we have completed [this deliverable] and thus we are moving in the correct direction for a successful project.
-- 13:12

Nathan: When are we delivering? When do we stop defining scenarios?
-- 13:14

Testing: How do we know we are within the speed limit on a given road? How could you know this without speed limit signs and a speedometer fitted to the car?
-- 13:15

Now we are on to Behat. Written in the Gherkin language, parsed by the Behat script library. After this, the behaviour of the feature is simulated and a report is automatically produced to say whether or not the feature behaved as expected.
-- 13:17

Step-by-step example of the Gherkin language presented on screen.
-- 13:18

Mink (open source acceptance test framework for web apps) - and other extensions, are used as a common gateway between Behat and the browser. This enables Behat to test defined features that are written in Gherkin language, in the web browser.
-- 13:22

Nathan is now running a series of automated acceptance tests - on 3G :) not ideal but the wifi is a little flaky right now. A couple of 'tuts' in the audience due to flaky internet.

He's now run through a few, and is demonstrating how straightforward it is to incrementally build up a testing library using Behat. So far he has been testing with Safari, and now he's put an additional argument to specifically use Chrome to test the Gherkin script. This looks great! Definitely want to start working with Behat when we get back to the UK.
-- 13:31

… s-l-o-w…. internet… Nathan: May be tricky to show some of the examples he had in store, specifically the JavaScript example.

The script he's been building is now automatically opening for the Wikipedia page, now on screen displaying the "Fish" page.
-- 13:34

Nathan is in awe of all the people that contribute so much time to develop and improve these kind of tools, for free. Also references the Dries keynote earlier, along the lines of the power of open source is in the passionate, talented community.
-- 13:36

Some scripting corrections to make the language used more descriptive relating to the feature, rather than being ambiguous. Previously the function was waiting for a designated period of time, rather than "when the suggestion box appears" - so he's updated this to make the test clearer.
-- 13:39

Behat enables you do test and complete projects in a more painless way. There will still be pain!
-- 13:44

Automatic deployment could be facilitated by using Behat to check before launching new features.

If it hurts, do it more frequently! Perform testing from the beginning, and do it continually throughout the project, in order to help stop a bottleneck occurring in the last stages of the project (and a bunch of stress for everyone!)
-- 13:47

Make sure that the specification is THE specification, and not a draft document that can be completed later. If this is done it ensures that the client signs off on the requirements in an unambiguous way.

There is a Jira extension, so that acceptance tests can be added to an issue-tracking and project management tool.

Tests can be tagged to not run (or not) on live sites.
-- 13:49

Behat assists the team in delivering what the client needs to a maximum level of quality.
-- 13:54
