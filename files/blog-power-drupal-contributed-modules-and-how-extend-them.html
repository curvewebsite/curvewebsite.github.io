The Power of Drupal - Contributed Modules (and how to extend them)

Craig Perks
Friday, 30th August, 2013

The following article will look at how Curve Agency extended an existing contributed module, SimpleAds, by adding in video advertisement functionality with reporting for a client who was using an advertising platform that wasn’t performing to a satisfactory level.

Foreword: Editing contributed modules is generally frowned upon within the Drupal community and should only be done as a last resort. By editing a contributed module, you’re more than likely destroying the upgrade path that would usually be available to you and if your code is written to a poor standard, you may leave yourself open to potential hacks.

Drupal is a pretty awesome resource. As standard, you get a secure, stable content management system, which can be extended with any of over 23,000+ contributed modules that add additional functionality not already included in the core Drupal install.

In very rare instances, a Drupal developer will find a contributed module that does some of what they want, but isn’t able to do everything. As a last resort in these instances it is possible to extend a contributed module and add in functionality missing from the original code.

Issues logging video impressions

A few months ago, an existing client of ours, QMJ Publishing, approached us and told us that they were unhappy with the existing advertising platform they were using, and that more specifically they found the success rate with which it was performing in delivering videos adverts to be pretty poor. The advertising platform they were using at the time was called OpenX.

The client was using the OpenX platform to deliver image based adverts on the same pages as the videos and noticed a massive difference between the number of video adverts successfully delivered versus the number of image adverts delivered and this was what initially set off the alarm bells at QMJ HQ.

Once we’d investigated we determined that the system wasn’t logging a large portion of the video impressions due to the amount of time people were spending on the page (in some instances under 10 seconds). The system was using an outdated Flash based version of FlowPlayer which often took this long or longer to load and delivered videos to the player as-uploaded without encoding them into a more suitable format. Additionally, these video ads were not loading on the majority of mobile, tablet and other devices that do not support Flash. This prompted us to look for a better solution that would fit the client’s needs more closely.

The requirements

After discussing the type of ads the client had, we determined that the system we ultimately chose needed to fulfill the following requirements:

  • Deliver video adverts ranging from 15 seconds to 10+ minutes in length
  • Report on total clicks, unique clicks, total impressions and unique impressions
  • Report per-video on the following events:
    • Play
    • Pause
    • Resume
    • Restart
    • Mute
    • Unmute
    • 50% Played
    • Finished
  • Deliver a weekly report email to a specified email address
  • Encode uploaded video content into file types suitable for use with a HTML5 video player over a variety of browsers (webm, mp4, ogg).

The Curve solution

Before specifically targeting a Drupal solution, we initially investigated a few off the shelf options to see if any of these would fit our requirements. What we found was that most video delivery solutions seemed to be geared more towards placing a video at the start of another video, the same way that YouTube delivers video adverts before giving you the content you actually wish to see. The cost of using many of these solutions seemed to be prohibitively high and so we looked at existing contributed Drupal modules to see if any of those could fit the bill.

We found a module called SimpleAds and although it didn’t allow for video delivery in the sandbox install we did, it did cover most of the other requirements and we were confident that we could add the functionality we needed into the module.

Modifying the module was relatively simple. We added an additional type called ‘Video Ad’ to the type dropdown, created a PHP template for this particular display type, added the necessary fields to the content type (video upload and placeholder graphic) and amended the module’s internal mechanics to work with the new advert type.

By default, SimpleAds creates 3 tables within Drupal when it’s installed ‘simpleads_blocks’, ‘simpleads_clicks’ and ‘simpleads_impressions’. We added a fourth table to this installation called ‘simpleads_video_stats’. This table was identical to the impressions table aside from the fact that it contained an ‘event’ column that was created to store the type of event being logged.

Fig 1. simpleads_video_stats table output from within phpMyAdmin

Early on we decided that we’d use the HTML5 version of FlowPlayer and installed this into the libraries directory of the Drupal install then tied this functionality into the SimpleAds module.

After getting the nuts and bolts of the module additions working, we still needed to encode the videos uploaded by QMJ and for our encoder of choice, we decided to go with FFMpeg which we then installed on the server.

After testing FFMpeg to ensure it worked outside of the Drupal environment we then integrated the functionality it offered us into the module. Knowing that some videos may take a reasonable amount of time to complete the encode, we added additional functionality to the module so that whilst a video was still awaiting encoding it could not be placed as a live advert on the site.

Fig 2. A list of uploaded videos also showing the ‘encode status’. Any videos with a red marker next to them cannot be set to ‘active’ on the site.

Extending the video ad player

We opted to use Flowplayer for video playback. The most recent version boasts strong features, is HTML5 powered with a Flash fallback, and has a powerful API. Flowplayer is also designed to be responsive, and given that the player needs to work on a number of devices in various positions, this feature was important when making the decision on what player to use.

The functionality available to us via JavaScript API were crucial as we needed to extend the interactivity of the player controls to allow us to track and log events.

Within the SimpleAds module itself we created a callback allowing us to track a variety of events when called from the player. We then used the Flowplayer API to bind this callback to the player controls, allowing us to log clicks and changes to the player via AJAX. These events would be tracked regardless of whether the user was being presented with the HTML5 or Flash player, or what device was being used.

Reporting on everything

The existing report pages that come with SimpleAds only store click and impression data, so we created an additional page specifically for video reporting. On this page we also list the other data specific to video plays (play, pause, resume, restart, mute, unmute, 50%+ played, finish). From this page a user can filter results by a date range. They’re also able to download the data as a CSV file which can be shown to advertising partners during meetings.

Fig 3. Video report page featuring date range filters and export to CSV link

In the few weeks since the module has been live on Agg-net.com the impression rate of videos has gone from approximately 35% to 95%, which confirmed our sneaking suspicions about the OpenX module.
