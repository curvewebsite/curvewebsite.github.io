A simple Drupal template selection system

Monday, 17th March, 2014

Use less content types and give 'basic page' the ability to change layout as required.

During the discovery phases of the responsive Relate charity website, the team decided that many content types identified were structurally similar. Therefore, a basic page content type was given the ability to change layout as required. In the following article, we will explore how this was implemented.

Required Layouts

After prototyping, we determined that some layouts would need to be available to the end-user (note: we’ve also included screenshots from the complete website to visualise the difference between the layouts here in the article).

  • Hero: This layout is intended as a top-level ‘call to action’ layout. On the hero layout the header image is large, full width and includes a blurb of supporting text.
  • Default: The default layout is the layout selected when a user initially clicks ‘Add content’ > ‘Basic page’. This layout is intended to be used for most pages throughout the site and is most useful when displaying pages of text-heavy content.
  • List: The list layout is functionally identical to the default layout aside from a block of links that appears on the bottom of the main body. These links are child-pages of the page you are on and they display the title and summary.
  • Rows: As with the list layout, the rows layout is structurally similar to the base default layout - the difference being the list of items that is displayed at the bottom of the main body. Unlike the list layout items, which are more akin to teasers, the rows layout items are more fully fleshed out displaying more information (such as the publication date).
  • Grid two columns/three columns: The grid two/three column layout is useful for pages that act as parent section pages within the site. These pages contain a short teaser followed by blocks of content - the content listed is child content of the page you are on.
  • Section primary/secondary: The section primary/secondary layouts were probably the most unusual layouts of the bunch. The intention with these layouts was to create the ability for articles to be split over several pages whilst also containing a table of contents at the top of every page and previous / next links at the bottom of the content.

Creation of the layouts taxonomy

The first step to implementing the layout functionality was to create the page layouts taxonomy, which simply contained a list of all the available layout types.

Next, using the Drupal ‘References’ module ( https://drupal.org/project/references) we created a term reference CCK field, which we attached to the basic page content type.

To place this item in its own vertical tab structure when adding a new piece of content, we used the appropriately named ‘Vertical Tabs’ module ( https://drupal.org/project/vertical_tabs).

Context layouts

At this point, when a user adds a new basic page, they can now click on the ‘Page settings’ vertical tab at the bottom of the page and select a layout. Admittedly doing so at this point wouldn’t actually achieve anything, but by adding the Drupal module Context into the mix ( https://drupal.org/project/context), we can begin to add the customised layouts dependent on particular conditions.

As touched upon previously a context will fire a particular layout based on conditions being matched (e.g. If you’re on a page with a node id of ‘node/1’ display a related content view display in the region ‘Sidebar Second’).

For our use case, we can leverage this functionality so that if a particular page layout is set, we can display specific blocks of output (e.g. the block of links at the bottom of the page on the list layout).

Note: It’s not touched upon in this article, but prior to creating a layout, we already had all the elements (usually blocks) created that were created to be embedded in the context.

Conclusion

As with any CMS there are several ways this particular behaviour could be accomplished within Drupal, but this is an effective way. If you find yourself working on a project which features lots of content administrators, and a requirement for layouts to be frequently changed, then this might be a good solution for you.

One more thing

Another facet of using a context driven system that we’ve not touched upon so far in this article is the fact that they can be chained together. For layouts that require some layout elements from another context (we discussed above how the default layout is used quite a bit in other layouts) this becomes a great addition as it reduces the amount of repetition.
